# Project Considerations

A candid review of the Hybrid Search Agent project: What works well, what trade-offs are baked in, what limitations to be aware of, and where there is clear room for improvement. Covers the parent project (retriever, FTS store, agent, dashboard) and the [`evaluation/`](evaluation/) subproject.

Items are framed as design notes for a learning/demo project rather than production blockers.

---

## Strengths

### Architecture and structure
- **Single source of truth for the FTS schema.** [`_FTS_COLUMNS`](fts_search.py) drives CREATE / INSERT / SELECT and the row → `SearchResult` mapping, so adding a column is a one-place edit.
- **Pydantic at every boundary.** [`ChunkMetadata`](pydantic_models.py), [`HybridSearchArgs`](pydantic_models.py), [`SearchResult`](pydantic_models.py) catch type errors before they reach the indexes or the agent.
- **Centralised configuration.** Model, search weights, FTS tuning, and debug flags all live in [`config.py`](config.py). README and dashboard reference it instead of duplicating values.
- **Layered documentation.** [`README.md`](README.md) for setup, [`info_project.md`](info_project.md) for the folder map, [`info_chroma_fts_metadata_chunks.md`](info_chroma_fts_metadata_chunks.md) for ingestion, [`evaluation/README.md`](evaluation/README.md) + [`evaluation/info_metrics.md`](evaluation/info_metrics.md) for the eval harness.

### Retrieval design
- **Hybrid retrieval is genuinely hybrid.** Per-backend score normalisation (BM25 capping, Chroma similarity, synthetic MMR rank) and configurable weights in [`HybridRetriever`](hybrid_search.py). Multi-mode FTS with weighted phrase + keyword + prefix in [`FTSStore.search_multi`](fts_search.py).
- **Live retrieval tuning.** Dashboard sliders mutate [`HybridRetriever`](hybrid_search.py) attributes directly, so weight changes take effect on the next call from either tab.

### Evaluation harness
- **Eval harness is self-contained.** Everything needed to grade the agent lives in [`evaluation/`](evaluation/) and runs as plain Python with no external grading service or database.
- **Metric registry as the single source of truth.** [`evaluation/eval_metric_registry.py`](evaluation/eval_metric_registry.py) defines every metric once: CSV columns, SQL schema, dashboard labels, and verdict gates all derive from it.

### Test coverage
- **134 passing unit tests** under [`evaluation/tests/`](evaluation/tests/), covering metric math, the registry, and eval helper plumbing.

## Trade-offs

- **SQLite FTS5 + Chroma instead of Postgres + pgvector, OpenSearch, Elasticsearch, Weaviate, or Redis.** [`fts_search.py`](fts_search.py) uses SQLite's built-in FTS5 virtual table for BM25 lexical search, [`hybrid_search.py`](hybrid_search.py) wraps a Chroma persistent collection for vector search, and per-backend score normalisation and weighted fusion live in [`HybridRetriever`](hybrid_search.py). Five common alternatives were considered and rejected. *Reason:* The project is a single-user, file-based learning/demo, and the chosen pair runs as plain Python with no daemon, container, or cluster. Every alternative trades that ergonomic for capabilities the project doesn't currently need.

  - **Operational footprint.** Both stores are file-based and open on import, nothing to install beyond `pip`, no server to start, no Docker. Postgres + pgvector requires a Postgres daemon (local install or container) plus the `pgvector` extension. OpenSearch and Elasticsearch both require a JVM with multi-GB minimum RAM and are cluster-oriented even single-node. Weaviate runs as a separate service, typically via Docker Compose. Redis Stack with RediSearch + Redis Vector requires the Redis server with the right modules loaded. For a notebook-scale project, every alternative adds an out-of-process dependency that the current pair avoids.

  - **Hybrid-search ergonomics.** Weaviate exposes BM25 + vector hybrid as one query API and would replace the per-backend normalisation and fusion code in [`HybridRetriever`](hybrid_search.py) entirely. OpenSearch and Elasticsearch can fuse via the same query DSL. Postgres requires writing the fusion in SQL or in Python, similar to today. Redis Stack supports hybrid queries but the API is less ergonomic than Weaviate's. The current pair pays the cost of writing and tuning fusion logic explicitly, which is also surfaced as a strength because every weight, cap, and rank rule lives in readable Python rather than in a vendor's query language.

  - **Lexical / BM25 quality.** Lucene-backed engines (OpenSearch, Elasticsearch) and Postgres `tsvector` are more sophisticated than SQLite FTS5 (richer analyzers, language-specific stemming, synonyms, custom tokenization). FTS5 supports unicode tokenization and trigram matching but is comparatively basic. Weaviate's BM25 is competent but less battle-tested than Lucene. Redis's full-text is similar in capability to FTS5. For the current corpus (small, mixed-domain documents) FTS5's defaults work. A domain-specific corpus needing custom analyzers would be a real reason to migrate.

  - **Vector-search quality.** Chroma uses HNSW internally and is competitive at small-to-mid scale. pgvector supports HNSW and IVFFlat with similar recall and latency at this scale. OpenSearch's KNN, Elasticsearch's `dense_vector` + KNN, Weaviate's HNSW, and Redis Vector all perform well. For typical notebook-size corpora no alternative would meaningfully change retrieval quality. At very large scales (tens of millions of vectors) dedicated vector DBs pull ahead, but that scale is not in scope here.

  - **Persistence and restart behavior.** Both current stores live as files in the project directory and survive restarts trivially — copy `chroma_db/` and `fts.db` to clone or back up the index. Postgres, OpenSearch, Elasticsearch, and Weaviate all require explicit dump/restore tooling. Redis defaults to in-memory and needs AOF or RDB configuration to behave like a durable store, and even configured, it competes with the host's RAM for working set, which is unsuitable for corpora that exceed memory.

  - **Future scale path.** When the project outgrows single-machine, the [`SearchResult`](pydantic_models.py) Pydantic boundary makes each backend interchangeable in principle: Swapping `FTSStore` for an OpenSearch client, or swapping Chroma for pgvector, only has to preserve the `SearchResult` shape that downstream code already depends on. Today's choice is intentional under-scaling: The cost is having to do that swap if/when scale demands it, and the gain is that nothing about the current single-machine workflow has to be paid for upfront.

- **Module-level side effects on import.** To support "import-and-go" usage in the Streamlit dashboard and REPL, [`hybrid_search_agent.py`](hybrid_search_agent.py) initializes the model, vector store, and retriever at the module level. While this simplifies the main entry points, it creates side effects during import. To prevent pathing errors in the evaluation suite [`evaluation/eval_deepeval.py`](evaluation/eval_deepeval.py), `sys.path.insert` + `chdir` is currently used as a localized patch. As the system scales, these components should be moved into an explicit build() factory to support cleaner testing and import-only contexts, and execuated from a separate main script.

- **Sidebar weight sliders mutate the shared retriever.** Moving the FTS or vector slider in the dashboard affects the agent's tool calls in the Chat tab simultaneously. Captioned in [`dashboard.py`](dashboard.py) but easy to forget. *Reason:* The Chat tab and the Eval tab share one retriever instance by design. Per-tab retrievers would let the two tabs silently disagree on weights, which is a worse failure mode for tuning than the documented surprise of one slider affecting both.
- **Single judge LLM.** [`evaluation/eval_config.py`](evaluation/eval_config.py) names one model for every DeepEval metric (no judge panel, no cross-judge validation). *Reason:* A judge panel multiplies DeepEval token spend per case by the panel size. For a learning/demo project where iterating on prompts and metric definitions dominates the cost picture, the marginal cross-judge signal is hard to justify against the bill.
- **SQLite ledger gates on full runs.** [`EvalLedger.save_run`](evaluation/eval_sqlite.py) only persists when every toggleable metric group is enabled. Partial runs land in JSON / CSV but not in the ledger. *Reason:* Keeps the dashboard's historical-trend view apples-to-apples. Pass-rate lines aren't polluted by partial runs whose gates were selectively switched off, at the cost of judge-only or routing-only iterations being invisible there until promoted to a full run.

- **Different options exist for the fixed-column schema in the evaluation harness. Current: The two-declarations + drift-assertion.** [`evaluation/eval_sqlite.py`](evaluation/eval_sqlite.py) keeps the fixed-column schema in two places: A literal `CREATE TABLE` string (`_RUNS_FIXED_PREFIX`/`_SUFFIX`, `_CASES_FIXED_PREFIX`/`_SUFFIX`) and a hand-maintained Python list (`_RUN_FIXED_COLS` / `_CASE_FIXED_COLS` / `*_TRAILING_COLS`). At import time, [`_assert_fixed_cols_align_with_sql`](evaluation/eval_sqlite.py) parses the SQL with a small tokenizer ([`_columns_in_sql_block`](evaluation/eval_sqlite.py)) and compares it to the lists, raising `RuntimeError` on any mismatch. Two alternatives exist: Derive the lists from the SQL using that same in-house tokenizer as a generator, or derive them using a mature SQL parser like `sqlparse`. *Reason for staying with the current pattern:* The schema is small and stable, and the duplication tax is tiny in practice. The verifier catches the most common drift case (developer edits the SQL but forgets the Python list, or vice versa) loudly, with a clear `in SQL: … / in lists: …` diff. **Caveat:** The tokenizer is in the loop for the assertion itself, so tokenizer bugs are *not* reliably caught. If the parser misses a column and the developer also forgot the hand-list, both sides agree on the wrong answer and the assertion passes silently, or if they disagree about a misread line, the diff message blames the side that is actually correct. So the hand-maintained list defends against developer drift, not against parser fragility. See the dedicated `sqlparse` upgrade bullet under `Possible future improvements`.

- **Ingestion notebook is a flexible template, not a polished pipeline.** Cells like `generate_summary`, `classify_content`, and `classify_doc` in [`chroma_fts_metadata_chunks.ipynb`](chroma_fts_metadata_chunks.ipynb) are intentional stubs to be adapted per corpus. *Reason:* Per-corpus document handling (summary style, classification taxonomy, doc-type rules) varies enough that a locked-down implementation would have to be ripped out for the next dataset. Intentional stubs are cheaper to fork than a polished pipeline is to undo.

## Limitations to note

- **Hardcoded `.env` path.** [`hybrid_search_agent.py`](hybrid_search_agent.py) and the ingestion notebook both call `load_dotenv()`.
- **Conversation state is in-memory only.** `InMemorySaver` in [`hybrid_search_agent.py`](hybrid_search_agent.py). The dashboard spins up a fresh thread per Streamlit session. Restarting the process loses everything.
- **Re-running the simple-add ingestion cells appends silently.** Use the upsert / refresh cells with `REFRESH_EXISTING_SOURCES = True` to overwrite stale data. Documented in the notebook header and in [`info_chroma_fts_metadata_chunks.md`](info_chroma_fts_metadata_chunks.md).
- **Dependencies are bleeding-edge.** `langchain==1.2.8`, `pandas==3.0.1`, `chromadb==1.5.7`, etc. Refresh pins carefully — any of these can break with a minor bump.

## Possible future improvements

- **Test coverage is metric-only.** No tests for [`hybrid_search.py`](hybrid_search.py), [`fts_search.py`](fts_search.py), [`pydantic_models.py`](pydantic_models.py), [`evaluation/eval_engine.py`](evaluation/eval_engine.py), [`evaluation/eval_sqlite.py`](evaluation/eval_sqlite.py), or [`evaluation/eval_report_manager.py`](evaluation/eval_report_manager.py). No parent-level `tests/` folder.
- **DBs are shared between dev and tests.** Tests run against the live `fts.db` / `chroma_db/`. A pytest fixture creating ephemeral stores per session would prevent contamination.
- **`run_agent()` is a 90-line dispatch loop** in [`hybrid_search_agent.py`](hybrid_search_agent.py) with deeply nested `if`/`elif` on event kind. Splitting into per-event handlers (`_handle_tool_start`, `_handle_chat_stream`, …) would make it more readable and easier to extend.
- **Single launcher entry.** Mixing module setup with the REPL in [`hybrid_search_agent.py`](hybrid_search_agent.py) means a second mode (scheduled batch, automation) would require restructuring. The CI agent's `agent.py` + `agent_modes.py` + `launch_*.py` split is the natural target if/when that need arises.
- **Dead code worth deciding on.** `PRINT` in [`config.py`](config.py) has no callers. `setup_logger_dual` in [`utils.py`](utils.py) is defined but never invoked. Either wire them up or remove.
- **Eval test cases are JSON, not parametric.** Adding many cases to [`evaluation/eval_cases.json`](evaluation/eval_cases.json) is verbose. A small case-builder helper would scale better.
- **Rename `expected_chunks` to `expected_snippets` in test cases.** The field in [`evaluation/eval_cases.json`](evaluation/eval_cases.json) holds representative *text snippets* used for substring matching, not chunk IDs, but the current name suggests the latter.
- **Multi-ledger DB browser/selector for the dashboard.** [`evaluation/eval_dashboard.py`](evaluation/eval_dashboard.py) reads from a single hardcoded `eval_ledger.db`. When a metric rename triggers a save-aside (per the Schema policy in [`evaluation/eval_sqlite.py`](evaluation/eval_sqlite.py)), the archived `eval_ledger_<YYYYMMDD>.db` becomes invisible to the dashboard. A small module that scans `evaluation/` for `eval_ledger*.db` files, exposes them in a sidebar dropdown, and lets the user pick which one to view would unlock the value of the save-aside workflow.
- **Replace the in-house schema tokenizer with `sqlparse` and drop the drift assertion.** Today the fixed-column schema in [`evaluation/eval_sqlite.py`](evaluation/eval_sqlite.py) is duplicated as a SQL literal and a hand-maintained Python list, with [`_assert_fixed_cols_align_with_sql`](evaluation/eval_sqlite.py) reconciling them at import time via [`_columns_in_sql_block`](evaluation/eval_sqlite.py), a 12-line best-effort tokenizer that is itself in the loop for every check. Because the tokenizer is part of the assertion, its bugs do *not* fail loudly in all cases, see trade-off "different options exist for the fixed-column schema in the evaluation harness" for further details. Pulling in `sqlparse` lets the SQL literal become canonical. Net effect: ~30 lines smaller, drift impossible by construction, and the tokenizer's parser-fragility false-protection goes away.
- **Backend labels are hardcoded literal strings duplicated across the codebase.** The five labels (`"fts"`, `"vector_similarity"`, `"vector_mmr"`, `"vector"`, `"hybrid"`) appear as raw literals in [`fts_search.py`](fts_search.py) (write site for `"fts"`), [`hybrid_search.py`](hybrid_search.py) (both as writes for the vector variants and as `==` comparisons inside `_hybrid_score`), the two duplicated frozensets in [`evaluation/eval_sqlite.py`](evaluation/eval_sqlite.py) and [`evaluation/eval_report_manager.py`](evaluation/eval_report_manager.py), the composite SQL column definitions in [`evaluation/eval_metric_registry.py`](evaluation/eval_metric_registry.py), and the test fixtures under [`evaluation/tests/`](evaluation/tests/). A typo in any one place silently breaks the chain, the row writes successfully on the production side but the eval misclassifies it into `backend_other` with no test catching it. A single-source option should be implemented istead.
